From CUDA Rays to OpenGL Pixels

Ray tracers are fun to write, but they only get truly satisfying once the result shows up on screen without the renderer taking the scenic route through system memory. CUDA and OpenGL both live on the GPU, but they do not automatically share resources, so displaying a CUDA-rendered image often turns into an awkward copy chain unless interop is set up properly [1][2][3].

This post covers the core idea behind the CUDA/OpenGL interop step: create an OpenGL texture, register it with CUDA, map it so CUDA can write into it, then hand it back to OpenGL for display [2][3]. It is one of those changes that looks like plumbing in the diff but feels like a major milestone in the renderer.

The Problem

Without interop, the image path usually looks something like this: CUDA renders into a device buffer, that buffer gets copied back to the CPU, and then OpenGL uploads the pixels again into a texture for drawing [2][3]. That workflow works, but it is exactly the kind of avoidable overhead that makes a GPU programmer stare into the distance for a while.

The better version is much cleaner: CUDA writes directly into an OpenGL-owned texture, and OpenGL displays that same texture on a fullscreen quad [2][3]. The whole frame stays on the GPU, which is where it wanted to be all along.

The Texture Becomes the Canvas

The first step is to create a regular OpenGL texture that will hold the rendered image. In practice, this is just a 2D texture with enough precision to store ray-traced color values, often something like GL_RGBA32F for float output.

glGenTextures(1, &glTexture);
glBindTexture(GL_TEXTURE_2D, glTexture);
glTexImage2D(
    GL_TEXTURE_2D,
    0,
    GL_RGBA32F,
    width,
    height,
    0,
    GL_RGBA,
    GL_FLOAT,
    nullptr
);

At this stage, the texture is just an OpenGL object. CUDA still has no idea it exists.

Let CUDA In

To make the texture accessible from CUDA, it gets registered as a graphics interop resource with cudaGraphicsGLRegisterImage [2][3]. That call is the official handshake between the two APIs.

cudaGraphicsResource* cudaResource;
cudaGraphicsGLRegisterImage(
    &cudaResource,
    glTexture,
    GL_TEXTURE_2D,
    cudaGraphicsRegisterFlagsWriteDiscard
);

The cudaGraphicsRegisterFlagsWriteDiscard flag tells CUDA that the texture will be overwritten rather than incrementally updated [2][3]. For a renderer that refreshes pixels every frame, that is exactly the right attitude.

Mapping the Resource

Registration alone is not enough. CUDA still cannot write into the texture until the resource is mapped for CUDA access [2][3].

cudaGraphicsMapResources(1, &cudaResource);

cudaArray_t cudaArray;
cudaGraphicsSubResourceGetMappedArray(
    &cudaArray,
    cudaResource,
    0,
    0
);

After this step, CUDA gets a cudaArray, which is its image-friendly view of the OpenGL texture [1][2][3]. This is the bridge that turns an OpenGL texture into something a CUDA kernel can target.

The Kernel Writes the Image

Once the mapped resource is available, the ray-tracing kernel can write colors directly into it. A typical pattern is one thread per pixel: compute screen coordinates, trace the ray, shade the result, and write the final color into the mapped image.

__global__ void renderKernel(
    cudaSurfaceObject_t surface,
    Scene scene,
    int width,
    int height
) {
    int x = blockIdx.x * blockDim.x + threadIdx.x;
    int y = blockIdx.y * blockDim.y + threadIdx.y;

    if (x >= width || y >= height) return;

    float3 color = traceRay(x, y, scene);

    surf2Dwrite(
        make_float4(color.x, color.y, color.z, 1.0f),
        surface,
        x * sizeof(float4),
        y
    );
}

That is the heart of the whole feature. Instead of producing an intermediate buffer and worrying about how to copy it around later, the kernel writes straight into the image that OpenGL will display.

Handing It Back to OpenGL

Once CUDA finishes rendering, the resource gets unmapped so OpenGL can safely use it again [2][3].

cudaGraphicsUnmapResources(1, &cudaResource);

That unmap step is the moment the texture changes hands. CUDA is done painting; OpenGL is now free to show the result.

Rendering it on screen is then refreshingly ordinary:

glUseProgram(shader);
glActiveTexture(GL_TEXTURE0);
glBindTexture(GL_TEXTURE_2D, glTexture);
glBindVertexArray(quadVAO);
glDrawArrays(GL_TRIANGLES, 0, 6);

The screen pass does not need to know anything about rays, BVHs, or whatever cinematic violence happened in the kernel. It just samples a texture and draws a quad.

Why This Commit Matters

This is a small architectural upgrade with outsized impact. It removes the GPU-to-CPU-to-GPU detour, reduces overhead, and makes the renderer feel much closer to a real interactive graphics application [2][3].

It also clarifies responsibilities nicely: CUDA computes the image, OpenGL presents it, and neither API has to pretend to be the other. That clean separation becomes even more valuable once accumulation, denoising, or live camera motion enters the picture.

A Useful Mental Model

The simplest way to picture the whole thing is this: OpenGL creates the canvas, CUDA paints on it, and OpenGL hangs it on the screen [2][3]. No unnecessary copies, no CPU babysitting, and no reason to move pixels off the GPU just to send them right back.

That is what makes this interop step feel so important. It is not just about showing the image; it is about making the renderer behave like a native GPU application instead of a demo stitched together with memory transfers.
